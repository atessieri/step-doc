======================
Command Line Interface
======================

Overview
========

The step_crash_prediction command line interface (CLI) provides a way to interact
with the STEP Crash Prediction.

The CLI allows users to:

* View the program version
* Generate JSON schema documentation for configuration
* Create configuration templates
* Load and run the module with a specific configuration

Basic Usage
===========

.. code-block:: bash

    step_crash_prediction [options]

If no options are provided, the program will display the help message.

Options
=======

The following command line options are available:

``-h, --help``
    Display the help message and exit.

``-v, --version``
    Show the program version and exit.

``-s, --conf_schema <schema file>``
    Write the configuration JSON schema to the specified file.

``-t, --conf_template <template file>``
    Write a configuration template to the specified file.

``-c, --config <config file>``
    Load the configuration from the specified file and run the module.

``-i, --instance <instance name>``
    Specify the instance name.

.. note::
    When the module uses the `-c` or `--config` option, also the -i or --instance option 
    must be specified to set the instance name

Detailed Command Description
============================

Help Command
------------

To display help information:

.. code-block:: bash

    step_crash_prediction --help

This will output usage instructions and a list of available options.

Version Command
---------------

To display the program version:

.. code-block:: bash

    step_crash_prediction --version

This will output the current version of the step_crash_prediction.

Configuration Schema Generation
-------------------------------

To generate a JSON schema file documenting all configuration options:

.. code-block:: bash

    step_crash_prediction --conf_schema /path/to/schema_file.json

This command creates a JSON schema file at the specified path. The file describes
all available configuration options, their types, defaults, and constraints. 
The schema can be used for validation or as a reference for creating custom configurations.

Configuration Template Generation
---------------------------------

To generate a configuration template file with default values:

.. code-block:: bash

    step_crash_prediction --conf_template /path/to/template_file.json

This command creates a JSON configuration file with default values 
for all settings. Users can modify this file as needed to customize their configuration.

Running with Configuration
--------------------------

To run the module with a specific configuration:

.. code-block:: bash

    step_crash_prediction --instance test --config /path/to/config_file.json

This loads the configuration from the specified file and starts the step_crash_prediction.
The module will connect to the Redis streams as specified in the configuration.

The configuration file is a JSON document that defines the behavior of the step_crash_prediction.

Exit Codes
==========

The program uses the following exit codes:

* ``0`` (OK): Successful termination
* ``1`` (USAGE): Command line usage error
* ``2`` (ERROR): General error during execution
* ``3`` (ABORTED): Operation aborted by user or system

Examples
========

Generate a configuration schema:

.. code-block:: bash

    step_crash_prediction --conf_schema ./schema.json

Create a configuration template:

.. code-block:: bash

    step_crash_prediction --conf_template ./template.json

Run with specific configuration:

.. code-block:: bash

    step_crash_prediction --instance test --config ./my_config.json

Generate both schema and template:

.. code-block:: bash

    step_crash_prediction --conf_schema ./schema.json --conf_template ./template.json

Program Operation
=================

When running with a configuration file, the step_crash_prediction
performs the following operations:

   #. Connects to the Redis server using the provided connection settings
   #. Initializes an ActorDataset for managing position trajectories with temporal data management
   #. Starts multiple asynchronous tasks based on configuration:

      - **Stream Message Reception**: Multiple concurrent Redis stream consumer instances (based on ``receiveClientNumber`` config) 
        continuously listen to Redis input streams for incoming CAM (Cooperative Awareness Messages), CPM (Collective 
        Perception Messages), and VAM (Vulnerable Road User Awareness Messages)
      - **Vehicle Position Processing**: Processes each received CAM message by converting it to position information, 
        inserting it into the actor dataset with cubic spline interpolation, and generating smooth position arrays
      - **Position Array Publishing**: When sufficient data is available, publishes interpolated position arrays 
        to Redis output streams for Crash Prediction and traffic monitoring systems
      - **Periodic Dataset Cleanup**: Automatically removes expired trajectory data at configurable intervals 
        (``cleanerInterval``) to prevent memory buildup and maintain optimal performance
   
   #. Automatically handles reconnection if connections are lost (using the specified reconnect interval)
   #. Manages graceful shutdown when interrupted with proper resource cleanup

The module operates as a real-time position processing pipeline, transforming raw GNSS measurements 
into smooth, regularly-sampled position trajectories optimized for Crash Prediction algorithms. It will continue 
running until manually interrupted (e.g., with Ctrl+C) or until a system signal is received.

Advanced Usage
==============

Signal Handling
---------------

The step_crash_prediction properly handles system signals for graceful shutdown. When a signal is received, 
it cleans up resources and terminates with the appropriate exit code.

Logging
-------

The step_crash_prediction includes detailed logging capabilities. Log settings can be configured 
in the configuration file, including:

* Log level (DEBUG, INFO, WARNING, ERROR, CRITICAL)
* Console logging
* File logging with rotation options

You can enable debug logging to troubleshoot connection or message handling issues.
