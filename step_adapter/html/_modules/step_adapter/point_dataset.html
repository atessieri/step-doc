  1"""Position dataset management with cubic spline interpolation.
  2
  3This module provides a comprehensive framework for managing position datasets
  4with advanced interpolation capabilities. It serves as a high-level interface for
  5processing GNSS tracking data, performing temporal validation, and generating smooth
  6position trajectories using cubic spline interpolation techniques.
  7
  8The module is designed for real-time tracking applications where raw GNSS
  9measurements need to be processed into smooth, regularly-sampled position data
 10suitable for analysis, visualization, or further processing. It handles common
 11challenges in GNSS data processing including temporal irregularities, data validation,
 12and cross-actor data contamination prevention.
 13
 14Architecture Overview:
 15    The module consists of two main components working in tandem:
 16
 17        #. **PointDatasetConfig**: Configuration and validation dataclass that defines
 18           processing parameters including path length, output frequency, and timing
 19           constraints.
 20        #. **PointDataset**: Main dataset manager that coordinates position validation,
 21           interpolation processing, and actor metadata management.
 22
 23    The architecture follows a layered approach where PointDataset provides a
 24    user-friendly interface while delegating core interpolation work to the
 25    underlying PointInterpolation system.
 26
 27Performance Characteristics:
 28    **Time Complexity Analysis** where n = input queue size, k = output points:
 29
 30        - **Point Addition**: O(n + k*log(n)) per point in typical cases where:
 31
 32            - O(n) = coefficient creation of spline cubic
 33            - O(k*log(n)) = generation of output points
 34
 35        - **Point Retrieval**: O(k) for result list with PositionInfo references
 36        - **Configuration Validation**: O(1) constant time operations
 37
 38    **Space Complexity Analysis** where n = input queue size, k = output points:
 39
 40        - **Input Storage**: O(n) for raw position points within path window
 41        - **Output Storage**: O(k) for interpolated position results
 42        - **Overall System**: O(n + k) with both n and m bounded by configuration
 43
 44    Memory Growth Characteristics:
 45        - **Bounded Growth**: Memory usage is limited by configuration parameters,
 46          not by total processing duration
 47        - **Predictable Footprint**: Maximum memory can be calculated from
 48          ``path_length_seconds`` and ``output_nominal_frequency``
 49        - **No Memory Leaks**: Automatic cleanup of outdated points prevents
 50          unbounded accumulation
 51
 52Key Features:
 53    - **Identity-Based Validation**: Ensures position data integrity by validating
 54      station and object identifiers before processing
 55    - **Cubic Spline Interpolation**: Generates smooth position trajectories
 56      between irregular GNSS measurements
 57    - **Configurable Output Frequency**: Supports flexible interpolation rates
 58      from 1Hz to higher frequencies
 59    - **Actor Metadata Management**: Tracks and updates actor dimensions
 60      alongside position data
 61    - **Temporal Consistency Validation**: Prevents processing of out-of-order
 62      or temporally inconsistent position data
 63    - **Gap Detection and Recovery**: Automatically handles large data gaps
 64      by clearing history to prevent unrealistic interpolation
 65    - **Resource Bounded Processing**: Guarantees predictable memory and
 66      computational resource usage
 67
 68Thread Safety:
 69    This module is not thread-safe. Applications requiring concurrent access
 70    from multiple threads must implement external synchronization mechanisms.
 71    Consider using thread-local storage or explicit locking strategies for
 72    multi-threaded environments.
 73
 74Performance Recommendations:
 75    - **Batch Processing**: For high-throughput scenarios, consider processing
 76      points in batches rather than individual calls
 77    - **Configuration Tuning**: Balance ``path_length_seconds`` and ``output_frequency``
 78      based on application requirements vs. computational resources
 79    - **Memory Monitoring**: Monitor memory usage in long-running applications,
 80      especially with high-frequency configurations
 81    - **Logging Levels**: Adjust logging verbosity in production to minimize
 82      I/O overhead from validation warnings
 83
 84Note:
 85    The interpolation quality is directly dependent on the density and temporal
 86    distribution of input GNSS points. Applications should ensure sufficient
 87    point coverage within the configured path length for optimal results.
 88
 89    Large temporal gaps exceeding the configured maximum deviation will trigger
 90    history clearing, which is necessary to prevent unrealistic interpolation
 91    across discontinuous data but may result in temporary output gaps.
 92"""

128    def __post_init__(self) -> None:
129        """Validate configuration parameters after dataclass initialization.
130
131        Performs validation on the following parameters:
132
133            - ``path_length_seconds``: Must be >= minimum required path length
134            - ``output_nominal_frequency``: Must be within allowed frequency range
135            - ``period_max_deviation_ms``: Must be >= 0 and < nominal period
136
137        Returns:
138            None
139
140        Raises:
141            ValueError: The exception raised if the configuration parameters are invalid.
142
143        Note:
144            The following conditions will raise a ValueError:
145
146                - ``path_length_seconds`` is negative.
147                - ``output_nominal_frequency`` is less than 1.
148                - ``period_max_deviation_ms`` is negative or greater than or equal to
149                  ``output_nominal_period_ms``.
150        """
151        if self.path_length_seconds <= 0:
152            raise ValueError("Path length must be greater than 0 seconds")
153        if self.output_nominal_frequency < 1:
154            raise ValueError(
155                "Output nominal frequency must be greater than or equal to 1"
156            )
157        output_nominal_period_ms = 1000 // self.output_nominal_frequency
158        path_length_ms = self.path_length_seconds * 1000
159        if path_length_ms <= output_nominal_period_ms * 2:
160            raise ValueError(
161                f"Path length {path_length_ms} ms must be greater than "
162                f"or equal to double the output nominal "
163                f"period {output_nominal_period_ms} ms"
164            )
165        if (
166            self.period_max_deviation_ms < 0
167            or self.period_max_deviation_ms >= output_nominal_period_ms
168        ):
169            raise ValueError(
170                f"Max deviation {self.period_max_deviation_ms} ms must be "
171                "greater than or equal to 0 and less than the nominal period "
172                f"{output_nominal_period_ms} ms"
173            )

176class PointDataset:
177    """A dataset manager for position tracking with interpolation capabilities.
178
179    This class provides a high-level interface for managing position data from a specific
180    station and object (actor), performing cubic spline interpolation to generate
181    smooth position trajectories at regular time intervals. It combines position
182    validation, temporal processing, and actor metadata management in a single
183    cohesive interface.
184
185    The PointDataset acts as a wrapper around the PointInterpolation system, adding
186    station/object identity validation and actor dimension tracking. It ensures data
187    integrity by rejecting points that don't match the configured station and object
188    identifiers, preventing cross-contamination between different tracking entities.
189
190    Key Features:
191
192        - **Identity Validation**: Ensures position data belongs to the correct
193          station and object before processing
194        - **Cubic Spline Interpolation**: Generates smooth position trajectories
195          between GNSS measurements
196        - **Actor Metadata Tracking**: Maintains actor length and width information
197        - **Configurable Output Frequency**: Supports customizable interpolation
198          output rates
199        - **Temporal Consistency**: Validates and maintains proper time sequencing
200          of position data
201
202    Performance Characteristics:
203
204        **add_point() Method Complexity** where n = input queue size, k = output points:
205
206            - **Time Complexity**: O(n + k*log(n)) in typical cases, O(1) for rejected points
207            - **Space Complexity**: O(n + k)
208            - **Dominant Operations**: Cubic spline coefficient calculation and interpolation
209
210        **get_points() Method Complexity** where k = number of interpolated points:
211
212            - **Time Complexity**: O(k)
213            - **Space Complexity**: O(k) for result list with PositionInfo references
214            - **Bounded Performance**: Configuration limits ensure predictable resource usage
215
216        **Overall System Complexity** where n = input queue size, k = output points:
217
218            - **Time Complexity**: O(n + k*log(n) + k)
219            - *Space Complexity**: O(n + k)
220
221    Thread Safety:
222        This class is not thread-safe. External synchronization is required if accessed
223        from multiple threads concurrently.
224
225    Attributes:
226        station_id_par (int): The station identifier parameter.
227        object_id_par (Optional[int]): The object identifier parameter. Can be None.
228        config (PointDatasetConfig): Configuration object containing dataset parameters
229            including path length in seconds, output nominal frequency, and period
230            max deviation in milliseconds.
231        logger (Logger): Logger instance for recording debug and info messages.
232
233    Example:
234        >>> config = PointDatasetConfig(
235        ...     path_length_seconds=5,
236        ...     output_nominal_frequency=5,
237        ...     period_max_deviation_ms=50
238        ... )
239        >>> dataset = PointDataset(
240        ...     station_id_par=123,
241        ...     object_id_par=456,
242        ...     config=config,
243        ...     logger=logger
244        ... )
245        >>> dataset.add_point(position_info, vehicle_length, vehicle_width)
246        >>> interpolated_points = dataset.get_points()
247
248    Note:
249        The interpolation quality depends on having sufficient input points spanning
250        the configured path length. Insufficient data will result in empty output
251        until adequate point history is established.
252
253        Large gaps in input data (exceeding maximum period deviation) will cause
254        history clearing to prevent unrealistic interpolation across temporal
255        discontinuities.

399    def add_point(
400        self,
401        position_info: PositionInfo,
402        vehicle_length: Optional[VehicleLength],
403        vehicle_width: Optional[int],
404    ) -> None:
405        """Add a new position point to the dataset and update actor dimensions.
406
407        This method validates incoming position data against the dataset's station and
408        object identifiers, converts the position timestamp to epoch milliseconds, and
409        delegates the point to the underlying interpolation system. It also updates
410        actor dimensions when provided.
411
412        The method performs the following operations:
413
414            #. **Identity Validation**: Verifies that the incoming position belongs to
415               the correct station and object IDs associated with this dataset.
416            #. **Time Conversion**: Converts ISO time format to epoch milliseconds
417               for internal processing.
418            #. **Point Delegation**: Forwards the validated point to the interpolation
419               system for processing and potential interpolation.
420            #. **Actor Metadata Update**: Updates actor length and width if new
421               values are provided.
422
423        Time Complexity:
424            O(1 + T_interpolation) where T_interpolation is the complexity of the underlying
425            interpolation system:
426
427            - **Validation Phase**: O(1) for station/object ID comparison
428            - **Time Conversion**: O(1) for ISO time to epoch milliseconds conversion
429            - **Interpolation Delegation**: Depends on PointInterpolation.add_point():
430
431                - **Best Case**: O(1) when validation fails or queue is filling
432                - **Typical Case**: O(n + k*log(n) + k) where:
433
434                    - n = input queue size in interpolation system
435                    - k = number of output points generated
436                    - Dominated by cubic spline interpolation calculations
437
438            - **Metadata Update**: O(1) for setting actor dimensions
439
440            **Overall**: O(n + k*log(n) + k) in typical successful cases, O(1) for
441            rejected points.
442
443        Space Complexity:
444            O(1 + S_interpolation) where S_interpolation is the space used by the
445            interpolation system:
446
447            - **Local Variables**: O(1) for validation tuples and time conversion
448            - **Interpolation System**: O(n + k) where:
449
450                - n = input queue size (bounded by path length configuration)
451                - k = output points stored in interpolation queue
452
453            - **Vehicle Metadata**: O(1) for storing length and width references
454
455            **Memory Impact**: No additional persistent memory allocation in this
456            method. All significant memory usage occurs within the delegated
457            interpolation system.
458
459        Args:
460            position_info (PositionInfo): Complete position information including
461                coordinates, speed, heading, station info, and ISO timestamp.
462                The stationId and objectId must match this dataset's identifiers.
463            vehicle_length (Optional[VehicleLength]): Actor length information.
464                If provided, updates the dataset's actor length. If None,
465                existing actor length is preserved.
466            vehicle_width (Optional[int]): Actor width in appropriate units.
467                If provided, updates the dataset's actor width. If None,
468                existing actor width is preserved.
469
470        Returns:
471            None
472
473        Note:
474            Points with mismatched station or object IDs are logged as warnings
475            and ignored without affecting the interpolation system. This prevents
476            data contamination across different vehicles or stations.
477
478            The ISO time conversion assumes the position_info.time field contains
479            a valid ISO 8601 formatted timestamp. Invalid timestamps will raise
480            exceptions during the conversion process.
481
482            Actor dimension updates are independent of position validation.
483            Even if a position is rejected due to ID mismatch, valid actor
484            dimensions will still be updated if provided.
485        """
486        station_info = position_info.stationInfo
487        current_ids = (station_info.stationId, position_info.objectId)
488        if current_ids != self._dataset_ids_tuple:
489            self._logger.warning(
490                msg=f"Point {current_ids} does not match the dataset {self._dataset_ids_tuple}"
491            )
492            return
493        position_time = ItsTime.create_from_iso_time(
494            time_str=position_info.time
495        ).epoch_ms
496        match self._point_interpolator.add_point(
497            position_time=position_time,
498            position=position_info,
499        ):
500            case (
501                PointInterpolation.InsertPointResult.SUCCESS
502                | PointInterpolation.InsertPointResult.FILLING_PERIOD
503                | PointInterpolation.InsertPointResult.PERIOD_TOO_LONG
504                | PointInterpolation.InsertPointResult.POSITION_ERROR
505            ):
506                pass
507            case PointInterpolation.InsertPointResult.PERIOD_TOO_SHORT | _:
508                return
509        if vehicle_length is not None:
510            self._vehicle_length = vehicle_length
511        if vehicle_width is not None:
512            self._vehicle_width = vehicle_width

514    def get_points(self) -> list[PositionInfo]:
515        """Retrieve all interpolated position points from the dataset.
516
517        This method extracts and returns all currently available interpolated position
518        data from the underlying interpolation system. The interpolated points represent
519        smoothed position values generated at regular time intervals based on the
520        configured output frequency.
521
522        The method performs a simple delegation to the interpolation system's
523        ``get_interpolated_values()`` method, which extracts position data from
524        the internal output queue containing (timestamp, PositionInfo) tuples.
525
526        Time Complexity:
527            O(k) where k is the number of interpolated points currently stored
528            in the output queue:
529
530                - **List Comprehension**: O(k) to iterate through all queue elements
531                - **Tuple Unpacking**: O(1) per element to extract PositionInfo
532                - **List Construction**: O(k) for creating the result list
533
534            The queue size is bounded by the configuration:
535            ``maxlen = (path_length_ms // output_nominal_period_ms) + 1``
536
537            **Typical Performance**: For standard configurations (e.g., 5-second path
538            at 5Hz frequency), n  26 elements, making this effectively O(1) in
539            practice due to the small, bounded queue size.
540
541        Space Complexity:
542            O(n) where n is the number of interpolated points:
543
544                - **Result List**: O(n) for storing PositionInfo references
545                - **Queue Access**: O(1) for accessing the underlying deque
546                - **Temporary Variables**: O(1) for list comprehension iteration
547
548            **Memory Characteristics**:
549
550                - **No Deep Copy**: Returns references to existing PositionInfo objects,
551                  not copies, minimizing memory overhead
552                - **Bounded Size**: Queue size is limited by configuration parameters,
553                  preventing unbounded memory growth
554                - **Immediate Return**: No additional processing or transformation,
555                  direct extraction from internal storage
556
557        Returns:
558            list[PositionInfo]: A list containing all currently available interpolated
559                position points. Each PositionInfo object contains complete position
560                data including coordinates, speed, heading, timestamp, and metadata.
561                The list is ordered chronologically with the oldest interpolated
562                point first.
563
564                **Empty List**: Returned when no interpolated data is available,
565                which occurs when:
566
567                    - Insufficient input points have been added for interpolation
568                    - The dataset was recently created or cleared
569                    - All points were rejected due to validation failures
570
571        Note:
572            The returned list contains references to the actual PositionInfo objects
573            stored in the interpolation system. Modifications to these objects will
574            affect the internal state. For read-only access, consider the returned
575            data immutable.
576
577            The interpolated points are generated based on cubic spline interpolation
578            of the input position data, providing smooth transitions between actual
579            GNSS measurements at the configured output frequency.
580
581            The temporal ordering of points in the returned list reflects the
582            chronological sequence of interpolated timestamps, which may not
583            necessarily match the order in which input points were added due to
584            potential timing irregularities in source data.
585        """
586        return self._point_interpolator.get_interpolated_values()
