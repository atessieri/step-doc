   2"""Multi-actor position dataset management with real-time interpolation.
   3
   4This module provides a comprehensive framework for managing position datasets across
   5multiple actors in real-time communication systems. It serves as the primary
   6interface for processing CAM (Cooperative Awareness Message) data from vehicle fleets,
   7providing thread-safe concurrent processing, automatic memory management, and high-performance
   8interpolation capabilities designed specifically for V2X (Vehicle-to-Everything) applications.
   9
  10The module implements a sophisticated multi-index architecture that efficiently scales with
  11fleet size while maintaining predictable performance characteristics and bounded memory usage.
  12It is designed to handle high-frequency telemetry streams from hundreds to thousands of
  13vehicles simultaneously, providing interpolated position trajectories suitable for real-time
  14analysis, collision avoidance, and traffic management systems.
  15
  16Architecture Overview:
  17    The module employs a layered architecture optimized for scalability and performance:
  18
  19        **Configuration Layer** (``ActorDatasetConfig``):
  20
  21            - Immutable dataclass providing system-wide parameter validation
  22            - Timing constraint enforcement and consistency checking
  23            - Performance tuning through configurable path length and output frequency
  24
  25        **Actor Management Layer** (``ActorDataset``):
  26
  27            - Multi-actor dataset coordination with O(log k) scaling
  28            - Thread-safe concurrent access for high-frequency CAM processing
  29            - Automatic lifecycle management and memory cleanup
  30
  31        **Interpolation Layer** (``PointDataset`` integration):
  32
  33            - Cubic spline interpolation for smooth trajectories
  34            - Temporal validation and error recovery mechanisms
  35            - Configurable output frequencies independent of input timing
  36
  37    This design enables the system to process CAM messages at rates exceeding 10,000 messages
  38    per second while maintaining microsecond-level latency and bounded memory usage.
  39
  40Key Components:
  41    **ActorDatasetConfig**:
  42        Configuration and validation dataclass that defines system-wide parameters for
  43        timing constraints, output frequency settings, and performance tuning. Provides
  44        comprehensive validation ensuring logical consistency between parameters.
  45
  46        - **Validation Complexity**: O(1) - Constant time parameter validation
  47        - **Memory Footprint**: ~64 bytes - Minimal overhead for frozen dataclass
  48        - **Thread Safety**: Immutable design ensures safe concurrent access
  49
  50    **ActorDataset**:
  51        Primary coordination hub managing multiple position datasets with
  52        sophisticated indexing for optimal performance across large vehicle fleets.
  53        Provides the main API for CAM message processing and position array generation.
  54
  55        - **Core Complexity**: O(n + m*log(n) + log(k) + p) for CAM processing
  56        - **Memory Scaling**: O(k*(n + m)) where k is fleet size
  57        - **Concurrency Model**: Thread-safe with logarithmic lock contention scaling
  58
  59Performance Characteristics:
  60    **Time Complexity Analysis** (System-Wide):
  61
  62        CAM Message Processing Pipeline:
  63            The complete processing pipeline exhibits the following complexity characteristics:
  64
  65                #. **Message Validation**: O(1) - CAM data structure validation
  66                #. **Position Conversion**: O(1) - Field mapping and object construction
  67                #. **Dataset Resolution**: O(1) - Hash table lookup for actor identification
  68                #. **Temporal Index Updates**: O(log(k)) - SortedList maintenance operations
  69                #. **Interpolation Processing**: O(n + m*log(n)) - Cubic spline calculations
  70                #. **Array Generation**: O(p) - Position array packaging and metadata assignment
  71
  72            **Overall Pipeline Complexity**: O(n + m*log(n) + log(k) + p) where:
  73                - n = input trajectory points per actor (bounded by path_length_seconds)
  74                - m = interpolated output points generated per actor
  75                - k = total number of tracked actors in the fleet
  76                - p = number of interpolated points retrieved for transmission
  77
  78        Fleet Scaling Performance:
  79            Scalability Characteristics:
  80                - **Primary Scaling Factor**: O(log(k)) with fleet size due to temporal indexing
  81                - **Interpolation Bound**: O(n + m*log(n)) limited by configuration parameters
  82                - **Memory Locality**: Optimized data structures for cache-friendly access patterns
  83                - **Branch Prediction**: Predictable conditional logic for pipeline optimization
  84
  85        **Cleanup and Maintenance Operations**:
  86            Automatic memory management operations designed for minimal impact on processing:
  87
  88                - **Best Case Cleanup**: O(log(k)) when no vehicles have expired
  89                - **Average Case Cleanup**: O(r*log(k)) where r << k for typical scenarios
  90                - **Worst Case Cleanup**: O(k*log(k)) during complete fleet turnover events
  91                - **Amortized Complexity**: O(log(k)) with regular cleanup scheduling
  92
  93    **Space Complexity Analysis** (System-Wide):
  94
  95        Memory Architecture:
  96            The module employs a multi-tier memory management strategy optimized for bounded
  97            growth and automatic cleanup:
  98
  99                Core System Storage:
 100                    - **Configuration Objects**: O(1) - Immutable parameter storage
 101                    - **Actor Registry**: O(k) - Hash table for actor dataset mapping
 102                    - **Temporal Index**: O(k) - SortedList for chronological ordering
 103                    - **Synchronization Primitives**: O(1) - Thread safety mechanisms
