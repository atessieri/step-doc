step_adapter.point_dataset module

Position dataset management with cubic spline interpolation.

This module provides a comprehensive framework for managing position datasets with advanced interpolation capabilities. It serves as a high-level interface for processing GNSS tracking data, performing temporal validation, and generating smooth position trajectories using cubic spline interpolation techniques.

The module is designed for real-time tracking applications where raw GNSS measurements need to be processed into smooth, regularly-sampled position data suitable for analysis, visualization, or further processing. It handles common challenges in GNSS data processing including temporal irregularities, data validation, and cross-actor data contamination prevention.

Architecture Overview:

The module consists of two main components working in tandem:

  1. PointDatasetConfig: Configuration and validation dataclass that defines processing parameters including path length, output frequency, and timing constraints.

  2. PointDataset: Main dataset manager that coordinates position validation, interpolation processing, and actor metadata management.

The architecture follows a layered approach where PointDataset provides a user-friendly interface while delegating core interpolation work to the underlying PointInterpolation system.

Performance Characteristics:

Time Complexity Analysis where n = input queue size, k = output points:

  • Point Addition: O(n + k*log(n)) per point in typical cases where:

    • O(n) = coefficient creation of spline cubic

    • O(k*log(n)) = generation of output points

  • Point Retrieval: O(k) for result list with PositionInfo references

  • Configuration Validation: O(1) constant time operations

Space Complexity Analysis where n = input queue size, k = output points:

  • Input Storage: O(n) for raw position points within path window

  • Output Storage: O(k) for interpolated position results

  • Overall System: O(n + k) with both n and m bounded by configuration

Memory Growth Characteristics:

  • Bounded Growth: Memory usage is limited by configuration parameters, not by total processing duration

  • Predictable Footprint: Maximum memory can be calculated from path_length_seconds and output_nominal_frequency

  • No Memory Leaks: Automatic cleanup of outdated points prevents unbounded accumulation

Key Features:

  • Identity-Based Validation: Ensures position data integrity by validating station and object identifiers before processing

  • Cubic Spline Interpolation: Generates smooth position trajectories between irregular GNSS measurements

  • Configurable Output Frequency: Supports flexible interpolation rates from 1Hz to higher frequencies

  • Actor Metadata Management: Tracks and updates actor dimensions alongside position data

  • Temporal Consistency Validation: Prevents processing of out-of-order or temporally inconsistent position data

  • Gap Detection and Recovery: Automatically handles large data gaps by clearing history to prevent unrealistic interpolation

  • Resource Bounded Processing: Guarantees predictable memory and computational resource usage

Thread Safety:

This module is not thread-safe. Applications requiring concurrent access from multiple threads must implement external synchronization mechanisms. Consider using thread-local storage or explicit locking strategies for multi-threaded environments.

Performance Recommendations:

  • Batch Processing: For high-throughput scenarios, consider processing points in batches rather than individual calls

  • Configuration Tuning: Balance path_length_seconds and output_frequency based on application requirements vs. computational resources

  • Memory Monitoring: Monitor memory usage in long-running applications, especially with high-frequency configurations

  • Logging Levels: Adjust logging verbosity in production to minimize I/O overhead from validation warnings

Note

The interpolation quality is directly dependent on the density and temporal distribution of input GNSS points. Applications should ensure sufficient point coverage within the configured path length for optimal results.

Large temporal gaps exceeding the configured maximum deviation will trigger history clearing, which is necessary to prevent unrealistic interpolation across discontinuous data but may result in temporary output gaps.

class step_adapter.point_dataset.PointDataset(station_id_par: int, object_id_par: int | None, config: PointDatasetConfig, logger: Logger)[source]

Bases: object

A dataset manager for position tracking with interpolation capabilities.

This class provides a high-level interface for managing position data from a specific station and object (actor), performing cubic spline interpolation to generate smooth position trajectories at regular time intervals. It combines position validation, temporal processing, and actor metadata management in a single cohesive interface.

The PointDataset acts as a wrapper around the PointInterpolation system, adding station/object identity validation and actor dimension tracking. It ensures data integrity by rejecting points that dont match the configured station and object identifiers, preventing cross-contamination between different tracking entities.

Key Features:

  • Identity Validation: Ensures position data belongs to the correct station and object before processing

  • Cubic Spline Interpolation: Generates smooth position trajectories between GNSS measurements

  • Actor Metadata Tracking: Maintains actor length and width information

  • Configurable Output Frequency: Supports customizable interpolation output rates

  • Temporal Consistency: Validates and maintains proper time sequencing of position data

Performance Characteristics:

add_point() Method Complexity where n = input queue size, k = output points:

  • Time Complexity: O(n + k*log(n)) in typical cases, O(1) for rejected points

  • Space Complexity: O(n + k)

  • Dominant Operations: Cubic spline coefficient calculation and interpolation

get_points() Method Complexity where k = number of interpolated points:

  • Time Complexity: O(k)

  • Space Complexity: O(k) for result list with PositionInfo references

  • Bounded Performance: Configuration limits ensure predictable resource usage

Overall System Complexity where n = input queue size, k = output points:

  • Time Complexity: O(n + k*log(n) + k)

  • Space Complexity*: O(n + k)

Thread Safety:

This class is not thread-safe. External synchronization is required if accessed from multiple threads concurrently.

add_point(position_info: PositionInfo, vehicle_length: VehicleLength | None, vehicle_width: int | None) → None[source]

Add a new position point to the dataset and update actor dimensions.

This method validates incoming position data against the datasets station and object identifiers, converts the position timestamp to epoch milliseconds, and delegates the point to the underlying interpolation system. It also updates actor dimensions when provided.

The method performs the following operations:

  1. Identity Validation: Verifies that the incoming position belongs to the correct station and object IDs associated with this dataset.

  2. Time Conversion: Converts ISO time format to epoch milliseconds for internal processing.

  3. Point Delegation: Forwards the validated point to the interpolation system for processing and potential interpolation.

  4. Actor Metadata Update: Updates actor length and width if new values are provided.

Time Complexity:

O(1 + T_interpolation) where T_interpolation is the complexity of the underlying interpolation system:

  • Validation Phase: O(1) for station/object ID comparison

  • Time Conversion: O(1) for ISO time to epoch milliseconds conversion

  • Interpolation Delegation: Depends on PointInterpolation.add_point():

    • Best Case: O(1) when validation fails or queue is filling

    • Typical Case: O(n + k*log(n) + k) where:

      • n = input queue size in interpolation system

      • k = number of output points generated

      • Dominated by cubic spline interpolation calculations

  • Metadata Update: O(1) for setting actor dimensions

Overall: O(n + k*log(n) + k) in typical successful cases, O(1) for rejected points.

Space Complexity:

O(1 + S_interpolation) where S_interpolation is the space used by the interpolation system:

  • Local Variables: O(1) for validation tuples and time conversion

  • Interpolation System: O(n + k) where:

    • n = input queue size (bounded by path length configuration)

    • k = output points stored in interpolation queue

  • Vehicle Metadata: O(1) for storing length and width references

Memory Impact: No additional persistent memory allocation in this method. All significant memory usage occurs within the delegated interpolation system.

Parameters:

  • position_info (PositionInfo) Complete position information including coordinates, speed, heading, station info, and ISO timestamp. The stationId and objectId must match this datasets identifiers.

  • vehicle_length (Optional[VehicleLength]) Actor length information. If provided, updates the datasets actor length. If None, existing actor length is preserved.

  • vehicle_width (Optional[int]) Actor width in appropriate units. If provided, updates the datasets actor width. If None, existing actor width is preserved.

Returns:

None

Note

Points with mismatched station or object IDs are logged as warnings and ignored without affecting the interpolation system. This prevents data contamination across different vehicles or stations.

The ISO time conversion assumes the position_info.time field contains a valid ISO 8601 formatted timestamp. Invalid timestamps will raise exceptions during the conversion process.

Actor dimension updates are independent of position validation. Even if a position is rejected due to ID mismatch, valid actor dimensions will still be updated if provided.

get_points() → list[PositionInfo][source]

Retrieve all interpolated position points from the dataset.

This method extracts and returns all currently available interpolated position data from the underlying interpolation system. The interpolated points represent smoothed position values generated at regular time intervals based on the configured output frequency.

The method performs a simple delegation to the interpolation systems get_interpolated_values() method, which extracts position data from the internal output queue containing (timestamp, PositionInfo) tuples.

Time Complexity:

O(k) where k is the number of interpolated points currently stored in the output queue:

  • List Comprehension: O(k) to iterate through all queue elements

  • Tuple Unpacking: O(1) per element to extract PositionInfo

  • List Construction: O(k) for creating the result list

The queue size is bounded by the configuration: maxlen = (path_length_ms // output_nominal_period_ms) + 1

Typical Performance: For standard configurations (e.g., 5-second path at 5Hz frequency), n 26 elements, making this effectively O(1) in practice due to the small, bounded queue size.

Space Complexity:

O(n) where n is the number of interpolated points:

  • Result List: O(n) for storing PositionInfo references

  • Queue Access: O(1) for accessing the underlying deque

  • Temporary Variables: O(1) for list comprehension iteration

Memory Characteristics:

  • No Deep Copy: Returns references to existing PositionInfo objects, not copies, minimizing memory overhead

  • Bounded Size: Queue size is limited by configuration parameters, preventing unbounded memory growth

  • Immediate Return: No additional processing or transformation, direct extraction from internal storage

Returns:

A list containing all currently available interpolated

position points. Each PositionInfo object contains complete position data including coordinates, speed, heading, timestamp, and metadata. The list is ordered chronologically with the oldest interpolated point first.

Empty List: Returned when no interpolated data is available, which occurs when:

  • Insufficient input points have been added for interpolation

  • The dataset was recently created or cleared

  • All points were rejected due to validation failures

Return type:

list[PositionInfo]

Note

The returned list contains references to the actual PositionInfo objects stored in the interpolation system. Modifications to these objects will affect the internal state. For read-only access, consider the returned data immutable.

The interpolated points are generated based on cubic spline interpolation of the input position data, providing smooth transitions between actual GNSS measurements at the configured output frequency.

The temporal ordering of points in the returned list reflects the chronological sequence of interpolated timestamps, which may not necessarily match the order in which input points were added due to potential timing irregularities in source data.
