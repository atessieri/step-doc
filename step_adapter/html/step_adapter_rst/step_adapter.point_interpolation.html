step_adapter.point_interpolation module

Point interpolation module for temporal and spatial trajectory smoothing.

This module provides a comprehensive solution for interpolating sparse position data into smooth, time-aligned trajectories. It is primarily designed for processing GNSS (Global Navigation Satellite System) data from tracking, navigation systems, and other location-based applications where temporal consistency and smooth motion representation are critical.

Key Components:

PointInterpolationConfig: Configuration dataclass for interpolation parameters PointInterpolation: Main interpolation engine with cubic spline-based smoothing

Features:

  • Cubic Spline Interpolation: Uses scipy CubicSpline for smooth spatial interpolation across latitude, longitude, and altitude dimensions

  • Temporal Validation: Enforces configurable minimum/maximum periods between points to maintain data quality and prevent unrealistic interpolations

  • Motion Parameter Calculation: Automatically computes speed and heading from position changes when missing from input data

  • Adaptive Output Generation: Produces time-aligned sequences at configurable nominal periods with tolerance for timing variations

  • Gap Detection and Recovery: Handles large temporal gaps by clearing interpolation history to prevent artifacts across data discontinuities

  • Memory-Efficient Processing: Uses deque-based sliding windows for optimal memory usage in continuous data streams

Performance Characteristics:

  • Time Complexity: O(n + k*log(n) + k) per point addition where n is input queue size and k is output points generated

  • Space Complexity: O(n + k) with bounded queues for steady-state operation

  • Memory Efficiency: Automatic cleanup of outdated data prevents unbounded memory growth during long-running operations

Typical Use Cases:

  • Vehicle trajectory smoothing for fleet management systems

  • Temporal alignment of multi-source position data

  • Generation of uniform sampling rates from irregular GPS updates

  • Real-time position interpolation for navigation applications

  • Motion analysis and parameter estimation from sparse position data

Dependencies:

  • scipy: For CubicSpline interpolation algorithms

  • numpy: For efficient numerical array operations

  • step_cloud_schemas: For standardized position data structures

  • collections: For deque-based queue management

Thread Safety:

Classes in this module are not thread-safe. External synchronization is required when accessing from multiple threads concurrently.

Example

Basic trajectory interpolation workflow:

>>> from step_adapter.point_interpolation import (
...     PointInterpolationConfig, PointInterpolation
... )
>>> import logging
>>>
>>> # Configure interpolation parameters
>>> config = PointInterpolationConfig(
...     path_length_ms=5000,        # 5-second sliding window
...     output_nominal_period_ms=100, # 10 Hz output rate
...     period_max_deviation_ms=20   # 20ms tolerance
... )
>>>
>>> # Create interpolator instance
>>> logger = logging.getLogger(__name__)
>>> interpolator = PointInterpolation(config, logger)
>>>
>>> # Process incoming position data
>>> for timestamp, position_info in position_stream:
...     result = interpolator.add_point(timestamp, position_info)
...     if result == PointInterpolation.InsertPointResult.SUCCESS:
...         interpolated_positions = interpolator.get_interpolated_values()
...         # Process interpolated positions...

A robust point interpolation system for temporal and spatial trajectory smoothing.

This class implements a sophisticated interpolation system designed for processing time-series position data, typically from GNSS devices. It creates smooth trajectories by interpolating between sparse position points using cubic spline interpolation, while maintaining temporal consistency and providing comprehensive motion parameter calculations.

The system operates on a sliding window approach, maintaining both input and output queues to handle continuous data streams efficiently. It performs temporal validation, gap detection, and automatic queue management to ensure data quality and prevent interpolation across unrealistic time gaps.

Key Features:

  • Cubic Spline Interpolation: Uses scipy cubic splines for smooth spatial interpolation across latitude, longitude, and altitude dimensions.

  • Temporal Validation: Enforces minimum and maximum time periods between points to maintain data quality and prevent unrealistic interpolations.

  • Motion Parameter Calculation: Automatically computes missing speed and heading information based on position changes and time differences.

  • Adaptive Output Generation: Produces time-aligned position sequences at configurable nominal periods with deviation tolerance.

  • Queue Management: Efficiently manages sliding windows of input and output data with automatic cleanup of outdated points.

  • Gap Detection: Identifies and handles large temporal gaps by clearing history to prevent interpolation across data discontinuities.

Algorithm Overview:

The interpolation process follows a multi-stage pipeline:

  1. Input Validation: Validates temporal consistency and rejects invalid points

  2. Queue Management: Maintains sliding windows of recent position data

  3. Spline Creation: Generates cubic splines for lat/lon/alt interpolation

  4. Output Generation: Creates time-aligned position sequences

  5. Motion Calculation: Computes speed/heading from position differences

Time Complexity Analysis:

Primary Operation - add_point(): O(n + k*log(n))

Where:

  • n = input queue size (typically bounded by path_length_ms / min_period_ms)

  • k = number of output points generated per operation

Detailed Breakdown:

  • Input validation: O(1) - constant time checks

  • Queue cleanup: O(m) where m = outdated points removed (typically small closed to 1 O(1))

  • Cubic spline creation: O(n) - linear in input points

  • Spline evaluation: O(k*log(n)) - k evaluations, each O(log(n)) binary search

  • Position object creation: O(k) - linear in output points

  • Motion calculations: O(k) - distance/bearing for each output point

  • Queue operations: O(k) - deque appendleft operations.

Best Case: O(1) - validation failures or insufficient data Average Case: O(n + k*log(n) + k) - successful interpolation (typical case) Worst Case: O(n + k*log(n) + k + n) - dense input requiring extensive cleanup (corner case)

Auxiliary Operations:

  • get_interpolated_values(): O(k) - linear scan of output queue

  • Property access (last_point_time): O(1) - constant time

  • Clear history: O(1) - deque.clear() is constant time

Space Complexity Analysis:

Overall Space Complexity: O(n + k)

Where:

  • n = maximum input queue size

  • k = maximum output queue size (bounded by maxlen parameter)

Detailed Memory Usage:

  • Input Queue: O(n) space for storing (timestamp, PositionInfo) tuples Typical size: path_length_ms / output_nominal_period_ms 50-100 points

  • Output Queue: O(k) space with automatic size management via maxlen Maximum size: (path_length_ms / output_nominal_period_ms) + 1

  • Cubic Spline Objects: O(n) space for polynomial coefficients Three interpolators (lat, lon, alt) each storing n-segment splines

  • Numpy Arrays: O(n) temporary space during spline creation Four arrays: times, latitudes, longitudes, altitudes

  • Geocode Object: O(1) constant space for distance/bearing calculations

  • Local Variables: O(1) constant space for iterators and temporaries

Memory Efficiency Features:

  • Automatic queue size limits prevent unbounded growth

  • Deque operations provide O(1) append/pop from both ends

  • Numpy arrays are reused and sized exactly to queue length

  • Old data automatically purged based on temporal windows

Steady-State Memory: In continuous operation, memory usage stabilizes at O(max_queue_sizes) regardless of total points processed over time.

Performance Optimizations:

  • Deque Data Structure: O(1) append/pop operations for queue management

  • Numpy Vectorization: Efficient array operations for coordinate processing

  • Binary Search Splines: O(log(n)) evaluation via scipys optimized implementation

  • Early Validation: O(1) rejection of invalid points before expensive operations

  • Bounded Queue Sizes: Prevents memory growth in long-running applications

  • Single-Pass Processing: Minimizes data traversal and copying operations

Typical Use Cases:

  • GNSS trajectory smoothing for tracking applications

  • Temporal alignment of position data for synchronized processing

  • Generation of uniform sampling rates from irregular position updates

  • Motion parameter estimation from sparse position data

  • Real-time position interpolation for navigation systems

Thread Safety:

This class is not thread-safe. External synchronization is required when accessing from multiple threads concurrently.

Note

The interpolation process requires a minimum of 4 points to create cubic splines. Points outside the configured temporal constraints are either rejected or cause history clearing to maintain interpolation quality. Speed calculations are scaled to 0.01 m/s units, and heading values are scaled to 0.1 degree units to match the expected data schema formats.

For optimal performance, configure path_length_ms to be 20-50 the output_nominal_period_ms to balance interpolation quality with memory usage.

add_point(position_time: int, position: PositionInfo) → InsertPointResult[source]

Add a new position point to the interpolation system and trigger output generation.

This method is the primary interface for feeding position data into the interpolation system. It validates the temporal consistency of incoming points, manages the input queue by removing outdated points, and triggers the interpolation process when sufficient data is available. The method implements several validation checks to ensure data quality and temporal coherence.

The algorithm performs the following operations:

  1. Empty Queue Handling: If the input queue is empty, adds the point and returns FILLING_PERIOD status.

  2. Temporal Validation: Checks if the new points timestamp is within acceptable bounds relative to the last point.

  3. Gap Detection: Identifies data gaps that are too large and clears history when necessary.

  4. Queue Management: Removes outdated points that fall outside the minimum path length window.

  5. Interpolation Trigger: Calls _update_output_queue() when sufficient data is available for interpolation.

Time Complexity:

The time complexity varies based on the validation outcome:

  • Best Case - Early Return: O(1) when validation fails (PERIOD_TOO_SHORT or empty queue scenarios)

  • Gap Detection Case: O(1) for clearing history and adding point (PERIOD_TOO_LONG)

  • Successful Case: O(n + k*log(n) + m) where:

    • n = current input queue size

    • k = number of output points generated by _update_output_queue()

    • m = number of outdated points removed from queue (typically small) so we can consider as O(1).

    The dominant factors are:

    • Queue cleanup: O(m) for removing outdated points. Typical we can consider as O(1).

    • _update_output_queue(): O(n + k*log(n)) for interpolation process

    • Deque operations: O(1) per append/popleft operation

Typical Case: O(n + k*log(n)) where k is proportional to the time span covered by new interpolated points.

Space Complexity:

O(n + k) where:

  • n = input queue size (bounded by path length configuration)

  • k = output points generated and stored in output queue

Memory Usage Details:

  • Input queue: O(n) for storing (timestamp, PositionInfo) tuples

  • Output queue: O(k) for interpolated positions (with maxlen limit)

  • Interpolator objects: O(n) for cubic spline coefficients

  • Local variables: O(1) constant space

  • Temporary objects: O(1) for validation and calculations

In steady state, both input and output queues have maximum size limits, making the practical space complexity O(max_input + max_output).

Parameters:

  • position_time (int) Timestamp of the position in milliseconds since epoch. Must be greater than the last points timestamp by at least _min_period_ms and less than _max_period_ms to maintain temporal consistency.

  • position (PositionInfo) Complete position information including coordinates, speed, heading, and metadata. The position will be validated and may trigger interpolation calculations.

Returns:

Enumeration indicating the result:

  • SUCCESS: Point added successfully and interpolation was performed. Output queue contains new interpolated positions.

  • FILLING_PERIOD: Point added but insufficient data for interpolation. More points needed to reach minimum path length requirement.

  • PERIOD_TOO_SHORT: Point rejected due to timestamp being too close to or older than the previous point. No changes made to queues.

  • PERIOD_TOO_LONG: Large temporal gap detected. History cleared and point added as new starting point for future interpolation.

Return type:

PointInterpolation.InsertPointResult

Note

The method maintains temporal ordering and data quality by rejecting points that would compromise the interpolation accuracy. When PERIOD_TOO_LONG is returned, the entire history is cleared to prevent interpolation across large data gaps that could produce unrealistic results.

The interpolation process is triggered only when there are sufficient points spanning the minimum path length, ensuring the cubic spline interpolators have adequate data for smooth curve generation.
